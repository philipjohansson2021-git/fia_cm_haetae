# HAETAE 결함주입: 공격·대응 상세 분석 (키복구 관점)

> 실제 HAETAE 전체 서명(x86, 공식 레퍼런스)에 라인별 결함을 주입한 시뮬레이션 결과를
> 바탕으로, **(A) 어디서·어떻게 비밀키가 복구되는가**, **(B) 각 대응기법이 어디에·어떻게
> 적용되어 무엇을 막는가**, **(C) 방지율을 어떻게 해석해야 하는가** 를 상세히 정리한다.
> 데이터: `campaign_raw.csv`(72조합), 키복구-유발 결함 47건. 코드: `sim_code/`.

---

## 0. 표기와 사전 지식

HAETAE 서명의 핵심 응답은 다음과 같다.

```
z = y + (−1)^b · c · s₁          (R_q = Z_q[x]/(x^256+1),  q = 64513)
```

| 기호 | 의미 | 공개/비밀 |
|---|---|---|
| `y` | 마스킹 nonce (하이퍼볼 샘플, 시드에서 유도, 고정소수점 스케일 Lₙ=8192) | 비밀(서명마다 새로) |
| `c` | 챌린지 다항식 (±1, 해밍무게 τ=58) | 공개(서명에 포함) |
| `s₁` | 비밀키 일부 | **비밀(공격 표적)** |
| `b` | bimodal 부호 비트 | 내부 |
| `Lₙ` | 고정소수점 큰 스케일(8192) | 공개 |
| `ĉ = NTT(c)` | c의 수론변환 | 공개로부터 계산 |

**키복구의 본질:** 응답 `z`에서 **마스킹 `y`를 제거**하면 `c·s₁`이 남고, `c`는 공개이므로
NTT 영역에서 `ĉ⁻¹`(성분별 역원)을 곱해 `ŝ₁`을 얻은 뒤 역변환하면 `s₁`이 복구된다.
즉 모든 공격의 공통 목표는 "**y의 제거 또는 예측**"이다.

**트레이스(trace) 수**: 키복구에 필요한 (정상/오류) 서명의 개수.
- **단일 트레이스(1)** = 결함 서명 하나만으로 즉시 복구 → 가장 치명적.
- **차분(2)** = 정상 + 오류 서명의 차이를 이용.
- **통계(다수)** = 여러 서명의 분포에서 점진적 복구.

## A. 오리지널(무방어) — 라인별 키복구 상세

아래 11개 라인 각각에서, 결함이 무엇을 바꾸고, 방출되는 `z`가 어떤 형태가 되며,
어떤 수식으로 `s₁`이 복구되는지, 몇 트레이스가 필요한지를 정리한다.
(괄호 안 숫자 = 해당 라인에서 키복구를 유발한 결함모델 수 / 시뮬 47건 분해.)

### A-1. `add_y` (T2) — +y 덧셈 스킵 **[단일 트레이스, 최고 위험]** (2건)
- **결함:** `z = y + (−1)^b c·s₁` 에서 **`+ y` 연산(덧셈 명령/함수호출)을 건너뜀.**
  단, y 자체는 메모리에 살아 있다(연산만 안 일어남).
- **방출 z:** `z = (−1)^b · Lₙ · (c·s₁)` — 마스킹이 통째로 빠짐.
- **복구:** 공개값 `c, Lₙ` 으로
  ```
  ŝ₁[k] = NTT(z / Lₙ)[k] · ĉ[k]⁻¹   (mod q),   s₁ = InvNTT(ŝ₁)
  ```
  부호 `(−1)^b`는 두 경우를 모두 시도해 노름이 작은 쪽을 택하면 결정된다.
- **검증:** 본 복구식은 HAETAE 레퍼런스 정수산술과 **비트 단위 768/768 일치**로 확인됨.
- **의의:** **단 한 번의 결함, 단 하나의 서명으로 `s₁` 전체가 즉시 복구**된다. 본 연구의 주표적.

### A-2. `y_sample` — 난수 y를 0으로 **[단일 트레이스]** (4건)
- **결함:** 하이퍼볼 샘플링 결과(또는 expandY)를 0/저엔트로피로 만듦(`y = 0`).
- **방출 z:** `z = (−1)^b c·s₁` — 효과적으로 **T2와 동일**.
- **복구:** A-1과 동일한 역연산. 단일 트레이스.
- **주의:** *결과적 z는 T2와 같지만 원인은 다르다*(여기선 y가 0, T2는 덧셈만 누락). 이 차이가
  대응기법(IRV) 탐지 여부를 가른다(아래 B-2 뉘앙스 참조).

### A-3. `seed_gen` — 시드 고정 → y 예측 가능 **[단일 트레이스]** (6건)
- **결함:** 샘플링 시드 생성(`xof(key‖rnd‖μ)`)을 스킵/0-고정 → 시드가 예측 가능한 값(공식
  구현은 초기 버퍼가 0)으로 고정 → 그 시드로 만든 **y를 공격자가 계산 가능**.
- **방출 z:** `z = y(예측) + (−1)^b c·s₁`.
- **복구:** 예측한 y를 빼고
  ```
  c·s₁ = (−1)^b (z − y),   ŝ₁ = NTT((z−y)/Lₙ) · ĉ⁻¹
  ```
  y를 알기 때문에 단일 트레이스로 복구.

### A-4. `cs1_mul` (T1) — c·s₁ 곱셈 스킵 **[2 트레이스]** (4건)
- **결함:** 곱셈 항 `c·s₁` 계산을 누락(`cs₁ = 0`).
- **방출 z:** `z ≈ y` — **nonce가 노출**됨.
- **복구:** 같은 nonce를 쓰는 정상 서명 `z_n = y + (−1)^b c·s₁` 과 결합:
  ```
  c·s₁ = (−1)^b (z_n − z),   ŝ₁ = NTT((z_n−z)/Lₙ) · ĉ⁻¹
  ```
  정상+오류 2 트레이스 필요(결정론 서명이면 동일 nonce 확보 가능).

### A-5. `sign_bit` — 부호비트 b 반전 **[2 트레이스 차분]** (6건)
- **결함:** bimodal 부호 `b`를 반전(또는 고정). 단일 오류 서명만으론 *유효한 다른-부호 서명*일
  뿐 직접 누설은 아니다(그래서 단일 트레이스 직접복구가 아님).
- **방출 z:** `z_f = y − (−1)^b c·s₁` (정상은 `z_n = y + (−1)^b c·s₁`).
- **복구(차분):**
  ```
  z_n − z_f = 2·(−1)^b c·s₁,   ŝ₁ = NTT((z_n−z_f)/(2Lₙ)) · ĉ⁻¹
  ```

### A-6. `unpackA` / `commit_Ay` / `lsb` / `challenge` — 챌린지 경로 오염 **[2+ 트레이스 차분]** (각 6/6/6/3건)
- **결함:** 공개행렬 언패킹(A), 약속 `A·y`, LSB 처리, 챌린지 c 등 **c가 결정되는 경로의 중간값을
  변조** → 커밋먼트와 불일치하는 `c′` 가 서명에 실림(검증 실패하는 오류 서명).
- **복구:** 정상 서명과 오류 서명의 **차분/선형 관계**(Lee 등의 분석)로 `s₁` 일부를 노출,
  다수 결합 시 전체 복구. 다중 트레이스.
- **시뮬 분류:** 이들은 `verify` 실패(`base_verify = −1`)로 나타난다 → 서명후검증이 탐지 가능.

### A-7. `reject_B1` (RB) — 거부 샘플링 우회 **[통계, 다수 트레이스]** (1건)
- **결함:** `if ‖z‖ ≥ B₁ then 재시작` 판정 분기를 스킵 → **서명경계 B₁을 벗어난 z**가 그대로 방출.
- **복구:** 경계를 벗어난 z들은 `s₁` 분포에 대한 추가 정보를 흘리며, **다수 수집 시 통계적/격자
  기법**으로 복구. 단일 트레이스는 아님.
- **특이점:** 검증경계 `B₂`가 생성경계 `B₁`보다 느슨(`B₁=9838.98 < B₂=12777.52`)하여,
  `‖z‖ ∈ [B₁, B₂)` 인 오류 서명은 **표준 검증을 통과**한다. → 서명후검증으로 못 잡는다.

### A-8. `key_mem` — 비밀키 영속 변조 **[2+ 트레이스 DFA]** (3건)
- **결함:** 메모리의 `s₁`을 지속적으로 변조(`s₁′ = s₁ + Δ`).
- **복구:** 변조 키로 만든 서명과 정상 서명의 **차분 오류 분석(DFA)** 으로 키 비트 복구.
- **특이점:** 변조가 *지속적*이라, 같은 입력을 두 번 서명해도 **두 실행 모두 동일하게 변조** →
  전체 이중연산이 두 출력을 비교해도 같아서 **탐지 실패**한다.

**오리지널 종합:** 무방어는 위 11개 라인 모두에서 키복구를 허용한다(**방지율 0% = 47/47 유발**).
가장 치명적인 것은 **단일 트레이스 직접복구(T2, y=0, seed)** 이며, 그중 **T2**는 "덧셈 한 번을
건너뛰는" 가장 단순한 결함으로 전체 키가 즉시 털린다.

> **하드웨어 확률과의 구분:** 본 시뮬은 "결함이 의도한 명령에 *정확히 들어갔다*"는 가정 하의
> 논리 결과다. 글리치가 실제로 그 명령에 떨어질 *확률*은 별개이며(STM32F3 실측에서 T2 단일
> 트레이스 키복구는 0.051% = 32/62,720 빈도로 재현됨, 다른 실험 참조), 본 분석은 "들어가면
> 복구되는가 / 대응이 막는가"에 집중한다.

## B. 대응기법 — 어디에·어떻게 적용되어 무엇을 막는가

### B-1. HAETAE-IRV
- **적용 위치:** 응답 `z`가 확정(거부 통과)된 **직후, 힌트 생성·인코딩 이전.**
- **동작(4 메커니즘):**
  - **M1 (재계산 검증):** `c·s₁`을 **독립적으로 다시 계산**(`InvNTT(NTT(c)∘ŝ₁)`)하고
    `z′ = y + (−1)^b c·s₁′` 를 만들어 방출된 `z`와 **일치하는지** 검사. 잔차
    `δ |= (z ≠ z′)`. → **연산 경로(곱셈·덧셈)가 깨진 결함**(T1, T2)을 탐지.
  - **M2 (서명경계 재검사):** `‖z‖²`을 독립적으로 다시 계산해 **생성경계 B₁**과 비교
    (`δ |= ‖z‖ ≥ B₁`). → **거부 우회(RB)** 탐지. (정상 수락 z는 `‖z‖<B₁`이라 오탐 없음.)
  - **M3 (키 체크섬):** `s₁`의 체크섬을 키 생성 시점의 기준값과 비교(`δ |= 체크섬 불일치`).
    → **비밀키 영속 변조(key_mem)** 탐지.
  - **M4 (무분기 감염):** 위 잔차 `δ`를 `PRF(δ)`로 출력 서명 전체에 **분기 없이 XOR 확산**.
    `δ=0`이면 서명 불변, `δ≠0`이면 서명이 난수화되어 무효. **"if 탐지 then 중단" 분기가
    없으므로**, 검사 분기를 두 번째 결함으로 스킵하는 우회가 불가능.
- **막는 라인(시뮬 측정):** `cs1_mul(T1)` 4/4, `add_y(T2)` 2/2, `reject(RB)` 1/1,
  `key_mem` 3/3 = **10/10 (100%)**.
- **못 막는 라인:** `seed`, `y_sample`, `sign_bit`, `unpackA`, `commit`, `lsb`, `challenge` = 0.

#### B-2. 핵심 뉘앙스 — IRV는 왜 "y=0"은 못 잡는데 "T2"는 잡는가?
둘 다 결과적으로 `z = c·s₁`(마스킹 없음)을 만든다. 그러나:
- **T2(덧셈 스킵):** y는 **살아 있는데** 더하지 않았다. IRV의 M1이 `z′ = y + c·s₁`(살아있는 y로)
  을 재계산하면 `z′ ≠ z(=c·s₁)` → **δ≠0 → 탐지.**
- **y=0(난수 0-고정):** y 자체가 0이다. IRV가 재계산해도 **같은 0인 y**를 쓰므로
  `z′ = 0 + c·s₁ = c·s₁ = z` → **δ=0 → 미탐.**
- **정리:** IRV는 *"응답 관계 z=y+(−1)^b c·s₁ 이 성립하는가"* 를 검증한다. **연산 무결성**은
  잡지만, **입력(y, c)의 품질·정합성**은 검증하지 않는다. 그래서 y/시드/부호/챌린지처럼
  *입력 자체가 오염*되어도 관계는 성립하는 결함은 IRV 범위 밖이고, 이는 난수경로 보호(아래
  Lee 등)가 담당한다.

### B-3. Lee 등 (JKIISC 2026)
- **적용 위치:** 시드 생성부 / 부호·난수 샘플부 / 서명 종료부.
- **동작:**
  - **정상성 검사(sanity):** XOF(난수) 버퍼가 전부 0이거나 동일 패턴이면 비정상으로 판정.
    → 시드 0-고정류 탐지.
  - **시드 독립 재유도:** `xof(key‖rnd‖μ)`를 한 번 더 계산해 사용한 시드와 비교. → 모든 시드 결함 탐지.
  - **부분 이중연산:** 부호비트·하이퍼볼 샘플을 재계산해 비교. → `sign_bit`, `y` 탐지.
  - **서명후검증(sign-then-verify):** 서명 후 표준 `Verify` 1회 수행. Verify는 커밋먼트에서
    `c′`을 재계산해 `c′==c` 와 **검증경계 B₂** 노름을 확인. → `unpackA/commit/lsb/challenge`
    (c′≠c)와 `T1/T2`(무효 z) 탐지.
- **막는 라인:** RB를 제외한 전부 = **46/47 (97.9%)**.
- **못 막는 라인:** **`reject(RB)`** — Verify가 *느슨한 B₂*로 검사하므로 `‖z‖∈[B₁,B₂)`인
  우회 서명이 통과한다(B₁<B₂). sanity·재계산도 노름을 안 보므로 RB를 못 잡는다.
- **비용:** 시드 재유도 + 하이퍼볼 **재샘플**이 비싸 오버헤드가 큼(x86 상대 ≈1.5×).

### B-4. 전체 이중연산 (Double Computation)
- **적용 위치:** 서명 함수 전체를 감싸는 래퍼.
- **동작:** 같은 입력으로 서명을 **2회 수행 후 출력 비교**, 불일치 시 폐기.
- **막는 라인:** *일시적(transient)* 결함은 한 실행에만 영향 → 두 출력 상이 → 탐지.
  = **44/47 (93.6%)**.
- **못 막는 라인:** **`key_mem`** — 메모리 변조가 *지속적*이라 두 실행 모두 동일하게 변조되어
  출력이 같다 → 비교로 탐지 불가(3건).
- **비용:** 서명 2회 → ≈2× (x86 상대 ≈1.7×).

| **방지율 (47건)** | 47 | **0%** | **21.3%** | **97.9%** | **93.6%** |

## C. 방지율(prevention rate) 해석 — 높을수록 좋다, 그러나 단일값만 보면 안 된다

### C-1. 정의와 방향
```
방지율 = (대응기법이 막은 키복구 결함 수) / (키복구를 유발하는 결함 수, 47)
```
**방지율은 높을수록 좋다.** 더 많은 키복구 공격을 차단한다는 뜻이므로 곧 더 안전함을 의미한다.
- **0%** = 무방어(모든 키복구 허용) = 최악.
- **100%** = 모든 키복구 차단 = 이상적.

### C-2. 단일 전체값의 함정
전체 방지율 하나만 보면 오해가 생긴다. 분모 47건은 **서로 성격이 다른 결함을 섞어** 놓았기
때문이다.
- **심각도(severity)가 다르다:** 단일 트레이스 직접복구(T2, y=0, seed) ≫ 2-트레이스 차분
  (T1, sign_bit, challenge…) ≫ 통계·다수(RB). "결함 하나·서명 하나로 전체 키"가 가장 위험하다.
- **방어 영역(domain)이 다르다:** IRV는 *연산 무결성*(응답 관계)을 지키는 기법이고, 난수경로
  결함은 *입력 품질* 문제로 영역이 다르다.

따라서 IRV의 **전체 21.3%** 는 "성능이 나쁘다"가 아니라, **분모에 IRV가 의도적으로 다루지
않는 난수경로 결함(37건)이 포함**되어 낮아 보이는 것이다.

### C-3. 공정한 비교 — class별 방지율

| 결함 class | 결함수 | **IRV** | Lee 등 | 이중연산 |
|---|---:|---:|---:|---:|
| **응답무결성·직접 키복구** (T1·T2·RB·key_mem) | 10 | **100%** | 90% | 70% |
| **난수·challenge 경로** (seed·y·sign_bit·unpackA·commit·lsb·challenge) | 37 | 0% | 100% | 100% |
| 전체 | 47 | 21.3% | 97.9% | 93.6% |

- **IRV:** 자신의 표적 class(직접 키복구·거부우회)에서 **100%**. 여기엔 (i) 가장 치명적인
  **단일 트레이스 직접 s₁ 복구(T2)**, (ii) 다른 두 기법이 못 막는 **거부우회(RB)** 가 포함.
  게다가 **무분기**(이차 결함 내성)이고 **오버헤드가 가장 낮다(≈1.0×).**
- **Lee 등:** 응답무결성 class에서 90%(**RB 1건 미방지**), 난수경로 100%. 단 **오버헤드 ≈1.5×.**
- **이중연산:** 응답무결성 class에서 70%(**영속 key_mem 미방지**), 난수경로 100%. **≈2× 비용.**

### C-4. 결론 — 어떻게 평가·보고할 것인가
1. **방지율은 높을수록 좋다**(키복구 차단율).
2. 그러나 **단일 전체값이 아니라 "class별 방지율 + 오버헤드"** 로 평가해야 공정하다.
3. **IRV**는 *가장 치명적인 직접 키복구(T2)와 거부우회(RB)를 최저비용·무분기로 100% 차단*
   하는 데 특화된 대안이고, **Lee 등**은 *난수경로를 넓게 막는* 대안이다(본 연구는 결합하지
   않고 각각 비교).
4. "키가 실제로 복구되는가"를 우선 지표로 두면, IRV가 막는 **단일 트레이스 직접복구·거부우회**
   가 가장 높은 우선순위의 위협이며, IRV는 이를 완전히 차단한다.
